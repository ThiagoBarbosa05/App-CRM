Corrija a autenticação da integração com a API V3 da Assertiva seguindo exatamente as instruções oficiais recebidas da empresa.

Erro atual:

Assertiva auth failed: 403 - {"message":"Invalid key=value pair (missing equal-sign) in Authorization header (hashed with SHA-256 and encoded with Base64): 'quA35IXMSoCREI27CRtsICi+4jWySsUIby+IEZwnpbY='."}

A autenticação oficial deve funcionar assim:

* Método: POST
* Endpoint: https://api.assertivasolucoes.com.br/oauth2/v3/token
* Tipo de autenticação: Basic Auth
* Usuário do Basic Auth: ClientID
* Senha do Basic Auth: Secret
* Corpo da requisição:
  grant_type=client_credentials
* O token retornado possui validade de 30 minutos.

Localize no projeto a função responsável pela autenticação da Assertiva e faça a menor correção possível.

O cabeçalho Authorization deve ser gerado com Base64 diretamente sobre:

ClientID:Secret

Exemplo correto em Node.js:

const clientId = process.env.ASSERTIVA_CLIENT_ID;
const clientSecret = process.env.ASSERTIVA_SECRET;

if (!clientId || !clientSecret) {
throw new Error(
"ClientID ou Secret da Assertiva não configurados nos Secrets do Replit."
);
}

const basicAuth = Buffer.from(
`${clientId}:${clientSecret}`,
"utf8"
).toString("base64");

const body = new URLSearchParams();
body.append("grant_type", "client_credentials");

const response = await fetch(
"https://api.assertivasolucoes.com.br/oauth2/v3/token",
{
method: "POST",
headers: {
Authorization: `Basic ${basicAuth}`,
"Content-Type": "application/x-www-form-urlencoded",
Accept: "application/json"
},
body: body.toString()
}
);

const responseText = await response.text();

if (!response.ok) {
throw new Error(
`Assertiva auth failed: ${response.status} - ${responseText}`
);
}

let data;

try {
data = JSON.parse(responseText);
} catch {
throw new Error(
"A Assertiva retornou uma resposta de autenticação que não é um JSON válido."
);
}

if (!data.access_token) {
throw new Error(
`A Assertiva não retornou access_token. Resposta: ${responseText}`
);
}

return data.access_token;

Importante:

1. Não aplique SHA-256 nas credenciais.

Não faça isto:

crypto
.createHash("sha256")
.update(`${clientId}:${clientSecret}`)
.digest("base64");

O Basic Auth exige apenas Base64 de:

ClientID:Secret

2. Não envie somente o Base64 no cabeçalho.

Errado:

Authorization: basicAuth

Correto:

Authorization: `Basic ${basicAuth}`

Deve existir um espaço entre “Basic” e o valor Base64.

3. Não envie o ClientID ou o Secret no corpo da requisição.

O corpo deve conter somente:

grant_type=client_credentials

4. Confirme que o corpo está sendo enviado como:

Content-Type: application/x-www-form-urlencoded

Não envie JSON, salvo se outra parte da documentação determinar isso.

5. Após receber o token, os endpoints de consulta da Assertiva devem usar:

Authorization: `Bearer ${accessToken}`

Não use Basic Auth nos endpoints de consulta. O Basic Auth é usado somente para gerar o token.

6. Salve as credenciais nos Secrets do Replit:

ASSERTIVA_CLIENT_ID
ASSERTIVA_SECRET

Caso o projeto já utilize nomes diferentes, como ASSERTIVA_CLIENT_SECRET, mantenha o nome existente e ajuste o código de maneira consistente.

7. Nunca exiba no console:

* Secret;
* cabeçalho Basic Auth completo;
* access_token completo;
* CPF ou CNPJ consultado.

8. Implemente cache do token, pois ele é válido por 30 minutos.

Exemplo:

let cachedToken = null;
let tokenExpiresAt = 0;

async function getAssertivaToken() {
const now = Date.now();

if (cachedToken && now < tokenExpiresAt) {
return cachedToken;
}

const tokenData = await solicitarNovoTokenAssertiva();

cachedToken = tokenData.access_token;

const expiresInSeconds =
Number(tokenData.expires_in) > 0
? Number(tokenData.expires_in)
: 1800;

tokenExpiresAt =
Date.now() + Math.max(expiresInSeconds - 60, 60) * 1000;

return cachedToken;
}

Renove o token aproximadamente 60 segundos antes da expiração.

A mensagem “Invalid key=value pair” sugere que o cabeçalho está sendo interpretado como outro modelo de autenticação porque o prefixo “Basic” está ausente ou o header foi construído incorretamente.
